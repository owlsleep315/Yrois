# Yrois — 코레일 승하차보조 / 유실물 관리 시스템

React + Vite + Electron 기반으로 제작된 **역무 운영용 데스크톱 애플리케이션**입니다.  
관리자 화면(`/admin`)과 역사 내 디스플레이 화면(`/display`)을 분리해, **입력/운영**과 **현장 표시**를 동시에 처리할 수 있도록 설계되었습니다.

## 프로젝트 소개

### 어떤 문제를 해결하나요?

기존 승하차보조 업무는 매우 아날로그적인 방식으로 운영되고 있었습니다.

승하차보조 요청이 들어오면 전화로 내용을 들으며 메모지에 먼저 기록하고,
이후 가시성을 위해 다시 칠판에 옮겨 적은 뒤,
마지막으로 인수인계용 기록지에 한 번 더 작성해야 했습니다.

즉, 하나의 승하차보조 건을 처리하기 위해 동일한 내용을 최소 세 번 반복 기록해야 하는 비효율이 존재했습니다.

또한 칠판 기반 운영 방식 특성상:
- 열차번호에 따른 도착시간을 직접 확인하기 위해 반복적으로 이동해야 했고
- 근무 책상과 칠판 사이를 오가야 하는 불편함이 있었으며
- 수기 작성으로 인해 글씨체가 제각각이라 가독성이 떨어지고
- 오기입 발생 시 수정테이프 등으로 직접 수정해야 하는 등
- 전반적으로 구식 운영 방식에 의존하고 있었습니다.
Yrois는 이러한 기존 운영 구조를 전산화하고 자동화하기 위해 제작되었습니다.

직원 및 사회복무요원이 각자 자리에서 키보드만으로 즉시 등록할 수 있도록 구성했으며,
관리자 입력 화면과 대형 display 화면을 분리하여 운영 효율성과 현장 가시성을 동시에 확보하는 것을 목표로 합니다.

### 기존 화이트보드 방식 대비 장점

- 수기 작성 대비 입력 형식 표준화 (열차번호, 도착시각, 분류, 담당자 등)
- 열차번호 기반 도착시간 자동 매핑으로 확인 실수 및 오기입 감소
- 기록 수정/삭제/완료 및 Undo/Redo 지원
- 도착 임박 열차(10분 이내) 시각 강조로 확인 누락 및 대응 실수 방지
- 기존 칠판 환경에서는 어려웠던 실시간 알림성 기능 구현 가능
- 대형 모니터 기반 display 화면으로 현장 가독성 강화
- 앱 재시작 후에도 기록 유지(JSON 저장)
- 월별 엑셀 파일 자동 생성으로 인수인계/정산 편의성 향상
- 운영 데이터를 장기적으로 보존하여 통계 및 분석 자료로 활용 가능

### 실제 사용 목적

- 직원이 메인 화면에서 승하차보조/유실물 건을 입력
- 다른 모니터(또는 별도 디스플레이)에 실시간 운영 현황 표출
- 월별 데이터 누적 관리
- 인수인계 자료 자동 생성 및 운영 기록 보존
- 향후 운영 통계 및 교통약자 이용 현황 분석 데이터로 활용

### 기대효과

- 반복적인 수기 업무 제거 및 운영 효율 향상
- 열차 도착시간 자동화 기반의 실수 감소
- 현장 가시성 향상 및 대응 누락 최소화
- 월별 교통약자 이용 현황 및 유실물 데이터 분석 가능
- 특정 역 단위 운영 도구를 넘어 **타 역으로 확장 가능한 표준 운영 템플릿** 확보
- 향후 통계 대시보드, 자동 보고서, 다중 역사 통합 운영 등 추가 기능 확장 기반 제공

## 기술 스택

### Core

- **React 19**
- **Vite 8**
- **Electron 42**
- **Electron Builder**

### Routing / Runtime

- **react-router-dom (HashRouter)**
  - `file://` 환경(패키징 앱)에서도 라우팅 안정성 확보

### Data / File

- **xlsx**
  - `train-data.xlsm` 열차시각표 읽기
  - 월별 운영 기록 `.xlsx` 자동 생성
- **Node.js fs/path**
  - JSON 영구 저장 및 디렉터리 생성

### Dev / Build Utilities

- **concurrently** (Vite + Electron 동시 실행)
- **wait-on** (개발 서버 준비 후 Electron 실행)
- **ESLint**

| 완료 처리 | 처리 완료된 건 숨김 처리 | `completed: true` 업데이트 |

| 실시간 화면 동기화 | 입력/수정 시 모든 창 즉시 반영 | `records:changed` IPC 브로드캐스트 |
| 엑셀 산출 자동화 | 변경 월 기준 운영 파일 자동 재생성 | `xlsx`로 월별 파일 작성 |
| Excel 열차시각표 연동 | 열차번호 입력 시 도착시간 자동 매핑 | `train-data.xlsm`의 `열차시각표` 시트 로딩 |
| 도착 임박 열차 강조 | 당일 기준 10분 이내 건 강조 표시 | `remainingMinutes <= 10` 스타일 |
| 전체화면 디스플레이 | 현장 표시 화면 몰입형 출력 | 보조 창 `fullscreen: true` |
| 날짜별 데이터 관리 | 이전/다음 날짜 이동 및 조회 | `dateKey(YYYY-MM-DD)` 기반 필터 |
| 포터블 데이터 저장 구조 | 실행 위치 기준 외부 폴더 저장 | 패키징 시 exe 경로 기준 `data/`, `exports/` |

## 실행 방법

### 1) 개발 환경 실행 (웹)

```bash
npm install
npm run dev
```

### 2) Electron 개발 실행

```bash
npm run dev:electron
```

### 3) 프로덕션 빌드

```bash
npm run build
```

### 4) 배포(포터블 디렉터리)

```bash
npm run dist:portable
```

빌드 후 `release/` 하위 결과물을 기준으로 실행할 수 있으며, 실행 환경에 따라 `data/` 폴더(기록/열차시각표)와 `exports/` 폴더(월별 산출 파일)가 함께 관리됩니다.

## 화면 구조

### `/admin` (관리자 화면)

운영자가 직접 입력/수정하는 메인 화면입니다.

- 날짜 이동(전일/익일/오늘)
- 승하차보조/유실물 등록 폼
- 목록 선택 후 수정
- 완료 처리(목록 숨김)
- 우클릭 삭제
- Undo / Redo

### `/display` (표시 화면)

현장 표시용 화면입니다.

- 상선/하선 분리 표기
- 도착 임박 항목 강조
- 데이터가 없을 경우 블랙아웃(번인 방지)
- 전체화면 표시

### 두 모니터 동작 방식

- 기본 모니터: `/admin` 창
- 보조 모니터: `/display` 창(전체화면)
- 보조 모니터가 없으면 동일 모니터에 두 창 생성
- 두 창 모두 동일 IPC 상태를 구독하여 즉시 동기화

## 데이터 저장 구조

### 1) `data/boarding-records.json`

운영 기록 원본 저장소입니다.

- 앱에서 등록/수정/삭제/완료할 때마다 즉시 갱신
- 레코드에는 `dateKey`, `trainNo`, `arrivalTime`, `boarding`, `type`, `completed` 등이 포함
- 앱 종료 후 재실행해도 데이터 유지

### 2) `exports/*.xlsx` (월별 자동 생성)

변경된 날짜가 속한 월 기준으로 파일 재생성됩니다.

- `YY년M월_승하차보조.xlsx`
- `YY년M월_유실물.xlsx`

`type` 값에 따라 자동 분류하여 각각의 파일에 기록됩니다.

### 3) `data/train-data.xlsm`

열차번호와 도착시각 매핑 소스입니다.

- `열차시각표` 시트 사용
- `열차번호`, `도착시각` 컬럼 기반으로 맵 생성
- 관리자 입력 시 열차번호 입력하면 도착시각 자동 입력

## 환경 및 배포 특징

### HashRouter를 사용하는 이유

패키징된 Electron 앱은 일반적으로 `file://.../index.html` 형태로 로드됩니다.  
브라우저 히스토리 기반 라우팅(BrowserRouter)은 새로고침/직접 진입 시 경로 해석 문제가 생길 수 있어, 해시 기반 라우팅을 사용합니다.

### `file://` 환경 대응

메인 프로세스에서 개발/배포 모드를 분기해 라우트를 로딩합니다.

- 개발: `http://localhost:5173/#/admin`, `#/display`
- 배포: `dist/index.html` + `hash` 옵션

### 포터블 실행 구조

패키징 빌드에서는 실행 파일 위치를 기준으로 데이터 루트를 계산합니다.

- 개발: 프로젝트 루트 기준
- 배포: `exe` 파일이 위치한 디렉터리 기준

따라서 앱과 함께 `data/`, `exports/` 폴더를 이동/백업하기 쉽습니다.

### data 폴더 외부 저장 이유

운영 데이터는 애플리케이션 코드(dist)와 분리되어야 업그레이드/재설치 시 손실 위험이 줄어듭니다.  
이 구조는 기록 유지와 인수인계에 유리합니다.

### Electron 단일 인스턴스 처리

중복 실행을 막고, 이미 실행 중이면 기존 창을 복구/포커스합니다.

### 듀얼 모니터 자동 감지

`screen.getAllDisplays()`로 디스플레이를 감지해

- 주 모니터: 관리자 창
- 보조 모니터: 표시 창(전체화면)

으로 자동 배치합니다.

## 운영 관점 가이드

- **일일 시작 전**: `data/train-data.xlsm` 최신본 확인
- **운영 중**: 관리자 화면에서 등록/완료 처리, 표시화면 실시간 확인
- **인수인계 시**: `exports/`의 월별 파일 공유
- **백업**: `data/boarding-records.json` + `exports/` 정기 백업 권장
